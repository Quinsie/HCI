1. 시스템 개요 · 테스트 모델

대학 강의용 전자출결 모바일 앱. 학생은 강의실 내에서 출석체크를 수행하고, 교수자는 실시간으로 출결 현황을 모니터링한다. 본 빌드는 실제 배포가 아니라 팀 내부에서 4개 핵심 Task를 검증하기 위한 단일 기기 테스트 빌드다.

1.1 운영자 프리셋(Operator Preset) 방식

실시간 교수–학생 연동이나 실제 GPS/네트워크 오류에 의존하지 않는다. 대신 진행자(운영자)가 앱의 숨겨진 디버그 화면에서 시나리오를 미리 세팅한 뒤 피험자에게 기기를 건넨다. 피험자는 그 프리셋대로만 앱을 사용한다. 이로써 지각·오류 등 엣지 상황을 재현 가능하게 만든다.

역할정의행위
운영자 (진행자)팀원. 테스트를 세팅·진행디버그 진입 → 프리셋 세팅 → 적용 → 핸드오프
피험자 (실험 대상)테스트 받는 사용자건네받은 앱을 프리셋대로 수행

2. 페르소나 · 계정

페르소나는 학생과 교수 두 종류만 둔다. (관리자 90000은 본 빌드 범위 밖 — 추후.)

학생2026000011q2w3e운영자가 자유 입력

  • 이름은 드롭다운이 아니라 텍스트 직접 입력(운영자가 작성). 기본값 비워두거나 예시 placeholder.
  • 로그인 화면은 항상 노출된다. 스플래시 3초 후 로그인 화면이 뜨고, 피험자가 로그인하면 시작한다. 프리셋이 있으면 해당 계정이 자동 입력될 수 있다.

3. 디버그 진입 · 운영자 프리셋

3.1 진입 (고정 제스처)

  1. 앱 최초 실행 → 스플래시(로고) 표시, 3초 정지.
  2. 스플래시에서 로고를 5번 탭하면 디버그 모드로 진입한다.
  3. 앱 작동 중에는 좌상단 '전자출결' 타이틀을 5번 탭해도 디버그로 진입한다(재부팅 불필요 · 재세팅/초기화용).
  4. 제스처는 고정이며 별도 해제 조건/PIN은 없다.
  5. 디버그에 진입하지 않으면 3초 후 로그인 화면으로 진행한다. 피험자가 로그인하면 저장된 프리셋대로 시작.

4. 권한 vs 환경 모델 (가장 중요)

출결 실패를 만드는 요인은 두 부류로 나뉘며, 피험자가 앱 안에서 고칠 수 있는지가 다르다.

4.1 권한 — 위치 · 블루투스 (피험자 변경 가능)

  • 운영자가 정하는 값은 어디까지나 초기값이다.
  • 초기값이 거부면, 피험자는 앱 내 권한 토글 버튼을 눌러 허용으로 바꿀 수 있다(실제 OS 권한 팝업 대신, 앱 위에서만 도는 가정 토글).
  • 권한이 거부인 채로 출결을 시도하면 권한 오류(PERM-03/BT-05)가 난다. 토글을 켜고 재시도하면 통과한다.

4.2 환경 — 강의실 · 네트워크 · 서버 (피험자 변경 불가)

  • 피험자는 이 값들을 바꿀 수 없다. 운영자만 설정한다.
  • 강의실 밖: 출석체크 자체를 차단한다(버튼 비활성). 오류가 아니라 "강의실 밖에서는 출석할 수 없음" 안내. (설계 전제: 강의실 밖 시도는 범위 밖.)
  • 네트워크 오류 / 서버 오류: 출결 시도 시 오류가 난다. 피험자가 못 고치므로 인증번호 fallback으로만 해결 가능(§7).
요인부류피험자 조치실패 시
위치 권한권한앱 내 토글로 허용오류 PERM-03
블루투스권한앱 내 토글로 허용오류 BT-05
강의실 밖환경불가차단 버튼 비활성
네트워크 오류환경불가 → 인증번호오류 NET-02
서버 오류환경불가 → 인증번호오류 SRV-04

5. 시간 모델

  • 인정 시간 = 5분 고정 (학생 프리셋에서 변경 불가, 표시용 상수).
  • 남은 인정 시간(잔여)은 운영자가 0:00~5:00 범위로 설정한다. 이는 피험자가 앱에 진입한 시점의 잔여 시간이다.
  • 앱 진입 후 잔여 시간은 실시간으로 카운트다운된다.
  • 출결 버튼을 누른 시점 기준: 잔여 > 0 → 출석 · 잔여 ≤ 0 → 지각.
예시
인정시간 5분, 남은시간 0:30으로 세팅 → 피험자는 "이미 4분 30초가 지났고 남은 30초 안에 처리해야 한다"는 상황에서 시작한다. 30초 안에 누르면 출석, 넘기면 지각.

운영자가 남은시간을 0:00(경과)로 두면 진입 즉시 지각 상황이 된다.

5.1 모든 시각은 기기 시계 기준으로 동적 계산

화면에 표시되는 시각·카운트다운은 하드코딩이 아니라 피험자 기기의 현재 시각에서 계산한다. 같은 프리셋이라도 테스트하는 실제 시점에 따라 숫자가 달라진다.

대상계산식예시
교수 수업 시간시작 = 현재 시각의 정시(시 단위 내림)
종료 = 시작 + 90분		now 22:12 → 22:00 ~ 23:30
학생 인정 window종료 = now + 남은시간
시작 = 종료 − 5분
남은 = 종료 − now		now 19:43, 남은 2:00 → 인정 19:40~19:45, 남은 02:00

6. 출결 판정 로직

학생이 출결 버튼을 누르면 아래 규칙을 위에서부터 평가하여 첫 번째로 매칭되는 결과를 적용한다.

#조건결과코드 / 화면피험자 복구
0강의실 밖차단— / 버튼 비활성불가
1위치 권한 거부오류PERM-03 / S07권한 토글 후 재시도
2블루투스 거부오류BT-05 / S07권한 토글 후 재시도
3네트워크 오류오류NET-02 / S07인증번호 fallback
4서버 오류오류SRV-04 / S07인증번호 fallback
5모두 정상 · 잔여 > 0출석S05—
6모두 정상 · 잔여 ≤ 0지각S06—

여러 결함이 동시에 있어도 위 우선순위의 첫 코드만 노출한다(정상 항목은 숨김). 피험자가 권한을 켜면 다음 우선순위가 재평가된다.

// 출결 버튼 onPress — 의사코드
function evaluateAttendance(env, perms, remainingSec, now) {
  if (env.classroom === '밖')      return { type:'BLOCKED' };       // 버튼은 애초에 비활성
  if (perms.location === '거부')     return { type:'ERROR', code:'PERM-03' };
  if (perms.bluetooth === '거부')    return { type:'ERROR', code:'BT-05' };
  if (env.network === '오류')       return { type:'ERROR', code:'NET-02' };
  if (env.server === '오류')        return { type:'ERROR', code:'SRV-04' };
  return remainingSec > 0
    ? { type:'PRESENT' }   // 출석
    : { type:'LATE' };     // 지각
}

7. 오류 처리 · Fallback

7.1 최초 시도 시각

학생이 처음 출결 버튼을 누른 시각을 저장한다. 이후 오류를 복구해 뒤늦게 처리하더라도 판정은 최초 시도 시각 기준으로 한다(기술적 문제로 인한 불이익 방지).

7.2 복구 수단

수단조건동작
재시도오류 화면(S07)현재 상태를 재평가. 운영자가 환경을 바꾸지 않았으면 같은 오류. 권한은 피험자가 켜면 통과.
인증번호 입력오류 발생 후에만교수/시스템이 자동 생성한 4자리 입력 → 성공 시 최초 시도 시각 기준 판정.
  • 인증번호는 독립 출석 수단이 아니다. 오류가 난 뒤에만 활성화되는 fallback.
  • 인증번호 성공 시: 최초 시도 시각의 잔여 > 0 → 출석(S13) · 잔여 ≤ 0 → 지각(S14).
  • 입력란 포커스 시 기기 기본 키패드(숫자)가 즉시 뜨도록 한다.

8. 학생 플로우 · 화면 동작

8.1 내비게이션 (하단 3탭)

홈 · 현황 · 설정 3개 탭으로 구성한다. 모든 학생 화면 하단에 고정 노출(출결 태스크 흐름 포함). 교수는 이 탭이 없으며 P01→P02→P09 단독 흐름이다.

8.2 홈

  • 최상단에 권한 체크리스트(위치 · 블루투스)를 두어 현재 켜짐/꺼짐을 ✓/✗로 표시한다. 출결 메인(S03)엔 별도 권한 배너를 두지 않는다.
  • 로그인 직후 진입. 현재 활성화된 출결(수업시간인 과목)을 가장 크게 강조해 바로 접근하게 한다 → 탭하면 출결 메인(S03).
  • 스크롤하면 나머지 수강 과목이 목록으로 보인다. 활성 아닌 과목은 회색 비활성, 탭하면 "수업시간이 아닙니다" 안내(진입 차단).
  • 홈에서 현황(사후확인 S10)과 설정으로 이동할 수 있다.

8.3 출결 진행 · 대기 큐

  • 출결 버튼을 누르면 버튼이 로딩으로 바뀌고 대기 큐가 생긴다. 큐는 시뮬레이션으로 약 7번부터 줄어들어 0이 되면 결과가 확정된다.
  • 정상·정시 → 출석 완료 / 인정시간 지남 → (로딩 후) 지각 처리 / 결함 → (로딩 후) 오류.

8.4 오류 & 복구

  • 인증번호는 고정값 7301.
  • 오류 해결은 설정에서 권한(위치/블루투스)을 켜고 재시도. (환경 오류는 피험자가 못 고치므로 인증번호로.)
  • 오류 상태 복원: 한 번 오류가 난 뒤에는 해당 과목에 다시 들어가도 출결 메인이 아니라 마지막 오류 화면(S07)이 다시 나온다.

8.5 설정

피험자가 바꿀 수 있는 권한 토글(위치 · 블루투스)을 둔다. 환경(강의실/네트워크/서버)은 노출하지 않는다(운영자 전용).

8.6 화면별 동작

화면ID동작 명세

일관성 원칙
S03(대기) → S04(진행) → S05/S06/S07(결과)은 동일 레이아웃에서 중앙 버튼의 색·텍스트만 바뀐다. 학생은 시선 이동 없이 결과를 인지한다.
핵심 UI 불변 원칙
학생 출결 화면(S03→S04→S05/S06/S07)의 중앙 원은 항상 같은 위치·같은 크기로 고정된다. 상태가 바뀌어도 원의 색과 텍스트만 바뀜다 — 파랑(출석체크) · 로딩 · 초록(출석) · 노랑(지각) · 빨강(오류). 위치 이동·크기 변화 금지(학생이 시선 이동 없이 결과 인지).

푸시 알림은 구현하지 않는다 — 테스크 자체가 출석 수행 시뮬레이션이므로 외부에서 진입하는 흐름이 없다.

9. 교수 플로우 · 검증 항목

교수는 환경 분기가 없다. 다음 3가지를 검증한다.

  1. 출석 시작 (P01): 인정 시간을 선택하고 해당 과목의 출석을 시작한다. 시작 시 4자리 인증번호가 자동 생성된다.
  2. 실시간 모니터링 (P02): 남은 진행 시간 동안 학생 대시보드의 카운트(전체·출석·지각·미처리·오류)가 제대로 증가한다. 더미 학생들이 출결하는 모습이 자동으로 찍힌다. (실시간 화면에 결석 탭 없음 — 미처리로 표시.)
  3. 상태 변경 (P09): 특정 학생을 선택해 출석/지각/결석으로 직접 변경할 수 있다(실행취소 포함).
더미 데이터
P02의 학생 출결 증가는 실제 학생 연동이 아니라 타이머 기반 더미 이벤트로 구현한다(예: N초마다 1~수명씩 출석 처리되어 카운트·리스트가 갱신).

9.1 대시보드 실시간 시뮬레이션

SCREEN 01에서 인정 시간을 고르고 출석을 시작하면, SCREEN 02 대시보드의 카운트가 타이머에 따라 더미로 변한다.

  • 시작 시점: 전원 미처리.
  • 인정 시간 동안: 대부분 출석으로, 소수가 오류로 전환(출석 비율이 훨씬 큼).
  • 인정 시간 종료 후: 남은 미처리 중 일부가 지각(또는 오류)으로, 끝까지 미처리면 수업 종료 시 결석 확정.
  • 따라서 "전원 출석/오류"가 되지 않는다. 실시간 화면엔 결석 탭 없이 미처리로 표시.
  • 명단 정렬: 오류 → 지각 → 미처리 → 출석 순, 같은 상태 안에서는 이름 오름차순(가나다). 시작 시 전원 미처리이며 시뮬레이션이 진행될수록 위 순서로 재정렬된다.

지난 강의 수정(과거 차시 정정)은 검증 Task 범위 밖으로 두고 구현하지 않는다. 상태 변경(P09)은 진행 중 세션의 특정 학생에 한한다.

교수 IA · 화면 범위
교수는 학생용 3탭(홈/현황/설정)을 갖지 않는다. 로그인 → 교수 홈(P-HOME, 오늘의 강의) → 강의 탭 → 출석 시작(P01, 인정시간 설정) → 대시보드(P02) → 상태 변경(P09)의 최소 흐름만 구현한다. 교수 홈은 학생 홈과 같은 형태로 '오늘의 강의'를 강조해 진입한다(권한 체크리스트·3탭 없음). 교수 사후 통계·과목 목록 등 추가 화면은 검증 Task 밖으로 생략한다.

10. 사후 확인 (Task 3)

  • 환경과 무관하게 항상 접근 가능한 공통 기능. 더미 데이터로 채운다.
  • 구조: 출석 현황 요약(S10) → 과목 선택 → 주차별 목록(S11, 아코디언 펼치기/접기) → 차시 상세(S12).
  • S10 상단에 "총 수업시수의 1/4 결석 시 F" 안내 + 위험도(정상/주의/위험).
  • 과목 카드에 전체 · 출석 · 지각 · 결석을 함께 표시한다. 전체 = 총 수업시수(15주 기준 총 차시)이며 현재까지 누계가 아니다 — 주 2차시 과목은 30, 주 1차시 과목은 15. (1/4 결석 = 전체÷4 기준으로 F 판정.)
  • S12(차시 상세)에는 처리 상태·처리 시각·최초 시도 시각·처리 방식·오류 발생 여부를 모두 표기 → 정정 요청 근거.
  • 색상: 출석 초록 · 지각 노랑 · 결석 빨강 · 미처리 회색 (색상+텍스트 병행, 색약 고려).

12. 데이터 · 영속성 모델

프리셋과 진행 상태는 기기 영속 저장소에 보관한다. 제안 스키마:

preset = {
  persona:      'student' | 'professor',
  name:         string,             // 자유 입력
  account:      { id, pw },         // 고정값
  subject:      string,             // 학생 전용
  acceptMin:    5,                  // 인정시간 고정
  remainingSec: number,             // 진입 시점 잔여(학생)
  perms:  { location: '허용'|'거부', bluetooth: '허용'|'거부' },   // 피험자 변경 가능
  env:    { classroom: '내'|'밖', network: '정상'|'오류', server: '정상'|'오류' } // 변경 불가
}

runtime = {
  firstAttemptAt: timestamp | null,  // 최초 시도 시각
  subjectStates:  { [subjectId]: 'idle'|'loading'|'present'|'late'|'error' },
  authCode:       '7301'             // 고정값
}
  • perms는 런타임에 피험자가 설정에서 토글로 갱신한다(초기값은 preset).
  • 과목별 상태(subjectStates)를 저장한다. error로 끝난 과목은 재진입 시 출결 메인이 아니라 마지막 오류 화면을 복원한다.
  • 핸드오프 후에도 위 상태는 유지되며, 운영자가 디버그에서 재설정할 때만 초기화된다.
